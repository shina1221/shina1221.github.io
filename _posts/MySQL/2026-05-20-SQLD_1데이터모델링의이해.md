# 정리  
모델링: 현실세계 단순화 한 것(약속된 표기법으로 DB 구축을 위한 분석/설계하는 과정)  
추상화(일정한 형식으로 표현), 단순화(복잡한 현실 세계를 제한된 표기법으로 표현), 명확화(이해하기 쉽게)  
모델링의 세가지 관점(데이터 관점(what), 프로세스 관점(how), 상관관점(process)//상관관점은 일의 방법에 따라 데이터가 어떻게 영향을 받고 있는지 모델링)  
모델링의 3단계(현실세계-> 개념적 모델링[ EA 수립 ] -> 논리적 모델링(관계중심[ key, 속성, 관계 ]) -> 물리적 모델링) // 중간에 낀 단계는 관계 중심  
*EA(Enterprise Architecture)란 전사적 아키텍처를 의미하며, 기업이나 조직의 비즈니스 목표와 IT 인프라를 일치시키기 위해 전체 구조를 설계하고 문서화하는 청사진  
프로젝트 생명주기(Life cycle)에서의 데이터 모델링  
정보전략계획(개념적 모델링) -> **분석(개념적 모델링, 논리적 모델링)** -> 설계(물리적 모델링) -> 개발 -> 테스트 -> 전환/이행  
데이터 베이스 3단계 구조  
외부 스키마(사용자 관점//각 사용자가 접근할 때 DB) -> 개념 스키마(통합 관점//DB에 저장되는 데이터 간 관계 표현) -> 내부 스키마(물리적) // 중간에 낀 단계는 관계 중심  
논리적 독립성(개념 스키마가 변경되어도 외부 스키마에는 영향x 지원), 물리적 독립성(내부 스키마가 변경되어도 외부/개념 스키마는 영향x 지원)=윗단계들에 영향 없이 지원  
만약 저장된 DB 구조가 바뀐다면 개념적/내부적 사상(매핑, 서로 다른 두 구조를 연결해서 대응시키는 것)이 바뀌어야함 그래야 개념 스키마가 그대로 남아있게 됨  
데이터 모델링의 3요소(업무 관여하는 어떤것(things), 어떤것이 가지는 성격(attributes), 업무가 관여하는 어떤것 간의 관계(relationships))  
데이터 모델링 용어  
어떤것(things): (엔터티(복수 개념), 인스턴스(개별 개념))  
어떤것 관의 연관(relationships): (관계(복수 개념), 패어링(개별 개념))  
어떤것의 성격(attributes): (속성(복수개념), 속성값(개별 개념))  
좋은 데이터 모델의 요소  
완전성, 중복 배제, 업무규칙(데이터 모델 분석만으로도 비즈니스 로직이 이해되어야함), 데이터 재사용(데이터 통합성과 독립성 고려), 의사소통, 통합성(동일 데이터는 유일학 정의해 다른 영역에서 참조)  
</br>  
엔터티:인스턴스의 집합, 업무에 필요하고 유용한 정보를 저장하고 관리하기 위한 집합적인 것 (Thing)  
ex)  
엔터티 : 과목 , 강사  
과목 : 수학 , 영어  
강사 : 이미소 , 조감독  
엔터티의 분류  
유무형에 따른 분류(유형, 개념, 사건)  
ex)유형(사원, 물품), 개념(조직, 상품), 사건(주문, 청구, 미납)  
발생 시점에 따른 분류(기본(키), 중심(메인), 행위(액션))  
ex)기본(고객, 상품), 중심(주문, 배송), 행위(주문변경이력)  
</br>  
속성:업무에서 필요로 하는 인스턴스에서 관리하고자 하는 의미상 더 이상 분리되지 않는 최소의 데이터 단위  
한 개의 엔터티(학생이 듣는 전체 강좌 집합)는 두 개 이상의 인스턴스(학생이 듣는 개별 강좌, 이름과 강좌를 합친 하나의 행)의 집합이어야 함  
한 개의 인스턴스는 두 개 이상의 속성(이름, 강좌 각각의 컬럼)을 가짐  
한 개의 속성은 한 개의 속성값(이미소, 영어같은 각각의 데이터)을 가짐  
속성의 분류  
기본속성(Basic Attribute)속성은 업무분석을 통해 바로 정의한 속성(제품번호, 제품명)  
**설계속성(Designed Attribute)**:원래 업무상 존재하지는 않지만 설계를 하면서 도출해내는 속성(제조사코드(LGC(LG그램12), APC(애플윙))))  
파생속성(Derived Attribute):다른 속성으로부터 계산이나 변형이 되어 생성되는 속성(판매금액 합계)  
**도메인:각 속성이 가질 수 있는 값의 범위(학점 도메인은 0에서 4.5사이의 값을 가짐, 주소는 길이가 20자 이내)**  
</br>  
관계: 엔터티 간 논리적인 연관성을 의미하고 존재에 의한 관계와 행위에 의한 관계로 구분  
관계는 엔터티 안에 인스턴스가 개별적으로 관계를 가지는 것(패어링)이고 이것의 집합을 관계로 표현  
존재에 의한 관계(부서 내 속해서 부서번호 생성), 행위에 의한 관계(주문번호)  
관계 차수(카디널리티):두 개의 엔터티 간 관계에서 참여자의 수를 표현하는 것  
관계명((Relationship//고객과 주문이라는 엔터티 기반_주문한다라는 행위), 관계차수((Relationship Degree//고객은 주문을 여럿 할 수 있다.),  
관계선택성((Relationship// 고객은 주문을 할 수도 안할 수도 있지만 주문은 반드시 고객정보를 포함해야한다. 필수적 비필수적), 관계정의((Definition// 고객은 여러 주문을 할 수 있으며 만드시 주문은 한 고객에 의해 생성된다, 업무규칙))  
</br>  
식별자:엔터티내에서 인스턴스들을 구분할 수 있는 구분자  
식별자의 특징  
유일성, 최소성(회원 ID만으로도 충분한데 이름+전화번호 이렇게 묶어서 식별자를 생성하지 않음), 불변성, 존재성(Null이면 안됨)  
식별자 분류  
대표성 여부:(주식별자(고객번호), 보조식별자(주민등록번호))  
*보조식별자:엔터티 내에서 각 행을 구분할 수 있는 구분자이나 대표성을 가지지 못해 참조관계 연결을 못함  
스스로 생성여부:(내부식별자(고객번호), 외부식별자(주문엔터티의 고객번호))  
속성 수:(단일식별자, 복합식별자(주문상세=주문번호+상세순번))  
대체여부(본질식별자(업무에 의해 생성(고객번호)), 인조식별자(고객번호+주문번호+순번))  
*인조식별자:업무적으로 만들어지지는 않지만 원조식별자가 복잡한 구성을 가지고 있기 때문에 인위적으로 만든 식별자  
식별자 관계만으로 연결된 데이터 모델의 특징은 주식별자 속성이 지속적으로 증가할 수 밖에 없는 구조로  
개발의 복잡성과 오류가능성을 유발시킬 수 있는 요인이 될 수 있음  
</br>  

# 헷갈리는 부분 정리  
</br>  
모델링과 스키마의 차이(=건축 설계와 설계도 기반 건물 뼈대)  
모델링(Modeling) = 현실 세계의 데이터를 어떻게 구조화할지 설계하는 과정  
무엇을 저장하고 어떤 관계가 있는가?  
ex)고객이 있다, 상품이 있다, 주문이 있다, 고객은 여러 주문을 할 수 있다.  
스키마(Schema) = 그 설계 결과를 실제 DB 구조로 정의한 것  
ex) create table에 정의하는 primary key라던가 number, date같은 속성을 지정해 주는 것  
</br>
DB매핑: 서로 다른 두 구조를 연결해서 대응시키는 것(=이 값은 여기랑 연결된다)  
ex)객체(DB테이블 매핑), 컬럼(변수 매핑), 관계(참조 연결)  
엔터티와 테이블 매핑(논리모델:고객 엔터티, 물리DB:CUSTOMER 테이블)  
속성과 컬럼 매핑(속성:고객명, 컬럼:CUSTOMER_NAME)  
관계와 외래키 매핑(고객ㅡ주문 관계)  
</br>
식별자(Identifier):각 데이터를 유일하게 구별할 수 있는 값  
유일성, 최소성(회원 ID만으로도 충분한데 이름+전화번호 이렇게 묶어서 식별자를 생성하지 않음), 불변성, 존재성(Null이면 안됨)  
ex)회원 ID(primary key)  
주식별자(사번), 보조식별자(유일하지만 대표는 아님//이메일, 주민번호), 내부식별자(시스템이 만든 값//AUTO_INCREMENT ID), 외부식별자(FK)  
식별자 관계  
식별자(부모PK가 자식PK에도 포함됨), 비식별자(부모PK가 자식FK로 존재)  
실무에서는 비식별자 관계를 주로 사용  
</br>  
엔터티(Entity):데이터로 저장할 객체  
ex)[김철수, 30, 남] 처럼 개인을 묶는 정보로 보통 고객이 엔터티가 됨  
모델링 기준 엔터티 = DB기준 테이블  
모델링 기준 속성 = DB기준 컬럼  
속성(Attribute): 엔티티가 가진 정보  
ex)이름, 나이, 성별 [김철수, 30, 남] 각 요소하나 하나가 속성이 됨    
</br>  
엔터티의 속성  
기본속성:원래 존재하는 값(이름, 생년월일)  
설계속성:시스템이 만든 값(회원번호, 주문번호)  
파생속성: 다른 값으로 계산되는 값(총 금액, 평균 점수)  
</br>  
카디널리티(cardinality): 관계의 개수 규칙  
엔터티 간 데이터가 몇개까지 연결될 수 있는가?  
ex)1:1, 1:1, N:M  
고객 1명, 주문 여러 건 ERD 표시  
고객 ─────< 주문  
   1       N  
DB에서는 FK로 지정하거나 중간 테이블을 만들어서 활용
